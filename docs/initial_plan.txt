I think we should use a hybrid migration strategy

Rather than introducing Alembic, Django migrations, or another framework immediately, I recommend we start with plain SQL migration files plus a very small Python migration runner that we own.

Why?

You already understand SQL deeply.
PostgreSQL-specific features (extensions, schemas, triggers, functions, indexes) are first-class citizens.
Every migration is readable and reviewable.
We avoid being constrained by an ORM.
Later, if we introduce Django models for part of the application, we can still keep the core Atlas database under explicit SQL control.

That approach gives us complete control over the database while keeping the migration system simple, transparent, and versioned from day one. I think it's the best fit for the kind of platform we're building.
