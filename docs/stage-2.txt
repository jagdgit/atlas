# If Atlas were my project, the next sprint would not be "Browser"

It would be **Job Engine & Planner**.

Because once you have that, everything else—browser plugins, YouTube processing, paper downloads, PDF readers, Python analysis, login pauses—becomes a capability the planner can invoke.

The browser is just one tool. The planner is what turns a collection of tools into an intelligent researcher.

## My proposed roadmap from here

1. **Job Engine** — persistent jobs, steps, status, pause/resume, artifacts.
2. **Planner** — decomposes objectives into executable plans.
3. **Tool Executor** — runs tools, captures outputs, retries, handles failures.
4. **Research Plugins** — web search, scholarly search, PDF ingestion, YouTube transcript retrieval, downloader.
5. **Evidence Graph** — every claim linked to its sources, confidence, and extracted evidence.
6. **Human-in-the-loop** — pause for login, CAPTCHA, manual downloads, or decisions, then resume.
7. **Report Generator** — produce a structured report with executive summary, methodology, evidence, confidence, references, and recommended next steps.
8. **Learning Pipeline** — optionally ingest the completed research into Atlas's knowledge base so future jobs start from what Atlas has already learned.

That architecture aligns with the system you've been describing all along: a persistent, deterministic research assistant that can tackle engineering problems like solar soiling estimation, prepare professional profiles, study technical standards, and build its own knowledge over time instead of simply generating one-off answers.

One final architectural recommendation

I would avoid building these as "browser tools" or "PDF tools." Instead, define each as a capability contract. For example:

SearchCapability
DownloadCapability
DocumentCapability
CodeCapability
ExecutionCapability
MemoryCapability
BrowserCapability

Then implement each capability with one or more plugins. Today SearchCapability might use DuckDuckGo and Semantic Scholar. Tomorrow you could add IEEE Xplore, NREL, or an internal document repository without changing the planner.

That separation keeps Atlas extensible and aligns well with your long-term goal of a deterministic research assistant that can work across engineering documents, source code, scientific literature, videos, and your own accumulated knowledge.
